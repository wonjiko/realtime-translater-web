# 검증 루프 워크플로우

피처 구현 완료 후, 독립된 서브에이전트들이 점수 기반 검증 루프를 수행한다.
각 에이전트는 독립된 컨텍스트에서 실행되며, `docs/features/` 파일을 통해서만 소통한다.

---

## 통과 기준

- **전체 평균 ≥ 8.0** AND **모든 항목 ≥ 5**
- 두 조건 모두 충족 시 PASS

## 정체 감지

- 현재 라운드 평균 - 이전 라운드 평균 ≤ 0.5이면 정체로 판단
- Round 1은 정체 감지를 건너뜀 (비교 대상 없음)
- 정체 시 사용자에게 선택지 제시:
  1. 저점수 항목 직접 수정 후 검증 재실행
  2. 해당 항목 통과 기준 완화
  3. 현재 상태로 수용

---

## 오케스트레이션 흐름

메인 세션이 오케스트레이터 역할을 수행한다. 각 단계에서 서브에이전트를 디스패치하고, 결과 파일을 읽어 다음 단계를 결정한다.

```
피처 요구사항 확정
  │
  ▼
① 플랜 에이전트 → {feature}-verification-plan.md (초안)
  │
  ▼
② 구현 에이전트 → 코드 변경
  │
  ▼
③ 플랜 보강 에이전트 → {feature}-verification-plan.md (보강본)
  │
  ▼
④ 검증 에이전트 → {feature}-verification-result.md (Round N)
  │
  ▼
판정 확인 ─── 평균≥8 AND 최저≥5 ──→ ✅ 통과
  │
  │ FAIL
  ▼
정체 감지 ─── 평균 변화 ≤ 0.5 ──→ ⚠️ 사용자 개입 요청
  │
  │ 평균 변화 > 0.5
  ▼
⑤ 수정 에이전트 → 코드 수정
  │
  └──→ ④ 검증 에이전트 (다시)
```

### 오케스트레이터 행동 상세

| 단계 | 메인 세션 행동 |
|------|---------------|
| ① 후 | `{feature}-verification-plan.md` 생성 확인. 파일 내용을 읽어 ②에 전달할 준비 |
| ② 후 | `git diff`로 변경 내용 캡처. 플랜 파일 내용과 함께 ③에 전달할 준비 |
| ③ 후 | 보강된 플랜 파일 내용 읽기. 관련 코드 파일 목록 파악하여 ④에 전달할 준비 |
| ④ 후 | 결과 파일에서 `전체 평균`, `전체 최저`, `판정` 파싱. PASS면 종료, FAIL이면 정체 감지 |
| 정체 감지 | Round N 평균 - Round N-1 평균 계산. ≤ 0.5이면 사용자 개입 요청 |
| ⑤ 후 | 수정 완료 확인. ④ 검증 에이전트 재디스패치 |

## 서브에이전트 프롬프트 템플릿

각 템플릿의 `{중괄호}` 부분은 오케스트레이터가 실제 값으로 치환하여 Agent 도구의 prompt 파라미터에 전달한다.

### ① 플랜 에이전트

```
역할: 피처 검증 플랜 설계자
목표: 주어진 피처 요구사항을 분석하여 검증 플랜을 생성한다.

## 입력
- 피처 요구사항:
{요구사항}

- 기존 코드 구조:
{코드구조}

## 지시
1. 요구사항을 5개 카테고리로 분해한다:
   - 기능 정합성 (Functional Correctness)
   - 에러 핸들링 (Error Handling)
   - 코드 품질 (Code Quality)
   - UX / 접근성 (UX & Accessibility)
   - 엣지 케이스 (Edge Cases)
2. 카테고리별 검증 항목을 도출한다
3. 항목마다 테이블 행을 작성한다: 번호, 항목명, 검증 방법, 10점 기준, 5점 미만 기준
4. 이 단계의 "검증 방법"은 예상 구현 기준으로 작성한다 (보강 단계에서 구체화됨)
5. docs/templates/verification-plan.template.md 의 포맷을 따른다

## 출력
- docs/features/{feature}-verification-plan.md 파일을 생성할 것
```

### ② 구현 에이전트

```
역할: 피처 구현자
목표: 검증 플랜을 인지한 상태에서 피처를 구현한다.

## 입력
- 피처 요구사항:
{요구사항}

- 검증 플랜:
{플랜내용}

## 지시
1. 검증 플랜의 각 항목을 인지하고, 높은 점수를 받을 수 있도록 구현한다
2. 구현 완료 후 자체 점검: 플랜의 5점 미만 기준에 해당하는 부분이 없는지 확인한다
3. 코드 변경만 수행한다. 검증 플랜 파일은 수정하지 않는다

## 출력
- 코드 변경 완료
- 변경 요약 (어떤 함수/파일을 추가/수정했는지)
```

### ③ 플랜 보강 에이전트

```
역할: 검증 플랜 보강자
목표: 실제 구현된 코드를 반영하여 검증 플랜의 검증 방법을 구체화한다.

## 입력
- 현재 검증 플랜:
{플랜내용}

- 코드 변경 diff:
{diff}

## 지시
1. 각 검증 항목의 "검증 방법"을 실제 코드 위치(파일:라인)로 구체화한다
2. 구현 과정에서 생긴 새로운 검증 포인트가 있으면 항목을 추가한다
3. 구현되지 않은 항목이 있으면 삭제하지 말고 그대로 유지한다 (검증 에이전트가 저점수를 매긴다)
4. 메타 섹션의 "보강" 날짜를 업데이트한다

## 출력
- docs/features/{feature}-verification-plan.md 파일을 업데이트할 것
```

### ④ 검증 에이전트

```
역할: 독립 검증자
목표: 검증 플랜에 따라 현재 코드를 채점한다. 편향 없이 엄격하게 평가한다.

## 입력
- 검증 플랜:
{플랜내용}

- 검증 대상 코드:
{코드내용}

- 라운드: {라운드번호}
- 이전 라운드 종합 점수 (있는 경우): {이전점수}

## 지시
1. 플랜의 모든 항목을 순서대로 검증한다
2. 각 항목에 1~10점을 채점하고, 근거를 1~2문장으로 기술한다
3. 10점 기준 / 5점 미만 기준을 참고하되, 자체 판단으로 중간 점수를 산정한다
4. 카테고리 요약 테이블을 작성한다 (카테고리별 평균, 최저)
5. 종합 판정을 계산한다:
   - 전체 평균 ≥ 8.0 AND 모든 항목 ≥ 5 → ✅ PASS
   - 그 외 → ❌ FAIL
6. FAIL인 경우 수정 가이드를 작성한다:
   - 5점 미만 항목: [필수] 태그
   - 8점 미만 항목: [권장] 태그
7. Round 2 이상이면 "이전 대비" 델타를 계산한다
8. 관대하게 채점하지 말 것. 의심스러우면 낮은 점수를 줄 것.

## 출력 포맷
docs/features/{feature}-verification-result.md 파일에 아래 형식으로 Round N 섹션을 추가한다.
(파일이 없으면 메타 섹션 포함하여 새로 생성한다)
docs/templates/verification-result.template.md 의 포맷을 따른다.
```

### ⑤ 수정 에이전트

```
역할: 품질 개선자
목표: 검증 결과의 저점수 항목을 개선한다.

## 입력
- 검증 결과의 수정 가이드:
{수정가이드}

- 검증 플랜:
{플랜내용}

- 현재 코드:
{코드내용}

## 지시
1. [필수] 항목을 먼저 처리한다 (5점 미만 → 5점 이상으로)
2. [권장] 항목을 이어서 처리한다 (8점 미만 → 8점 이상 목표)
3. 수정 시 다른 항목의 점수를 떨어뜨리지 않도록 주의한다
4. 검증 플랜, 검증 결과 파일은 수정하지 않는다. 코드만 수정한다.

## 출력
- 코드 수정 완료
- 수정 내역 요약 (항목별로 무엇을 변경했는지)
```
