# 문서 생성 룰

이 문서는 프로젝트 기능 개발 시 작성해야 하는 4가지 문서 타입의 작성 규칙을 정의한다.

## 문서 라이프사이클

기능 단위로 아래 사이클을 따른다:

```
PRD (요구사항 정의) → 구현 → TESTING (테스트 계획) → TESTING-REPORT (검증 결과) → PLAN (개선 계획) → 구현
```

### LLM 기반 워크플로우

| 단계 | 입력 | LLM 작업 | 출력 |
|---|---|---|---|
| 1 | PRD.md | 코드 구현 | 소스 코드 |
| 2 | TESTING.md + 소스 코드 | 검증 수행 | 검증 결과 (raw) |
| 3 | 검증 결과 + PRD.md | 리포트 작성 | TESTING-REPORT.md |
| 4 | TESTING-REPORT.md | 개선 계획 수립 | PLAN.md |
| 5 | PLAN.md | 수정 구현 | 소스 코드 (개선) |

## 공통 규칙

### 파일 구조

```
docs/
├── RULES.md                    # 이 파일 (문서 생성 룰)
├── templates/                  # 빈 템플릿 (복사해서 사용)
│   ├── PRD.template.md
│   ├── TESTING.template.md
│   ├── TESTING-REPORT.template.md
│   └── PLAN.template.md
├── v1/                         # 최초 릴리즈 문서 (예시)
│   ├── PRD.md
│   ├── TESTING.md
│   ├── TESTING-REPORT.md
│   └── PLAN.md
└── features/                   # 기능별 문서 폴더
    └── {feature-name}/
        ├── PRD.md
        ├── TESTING.md
        ├── TESTING-REPORT.md
        └── PLAN.md
```

### 네이밍 규칙

- **폴더명**: 기능을 나타내는 kebab-case (예: `offline-mode`, `retry-logic`, `accessibility-improvement`)
- **파일명**: 고정 — `PRD.md`, `TESTING.md`, `TESTING-REPORT.md`, `PLAN.md`
- 4개 문서는 항상 같은 폴더에 위치

### 작성 원칙

1. **언어**: 한국어 기반 + 영어 기술 용어 혼용
2. **구조화**: LLM이 파싱 가능한 포맷 — 테이블, 코드블록, 번호 매긴 리스트, 체크리스트
3. **상호 참조**: 같은 폴더 내 문서 간 상대 경로로 링크 (예: `[PRD](./PRD.md)`)
4. **구체성**: 모호한 표현("적절한", "빠른", "많은") 금지 → 구체적 수치/조건 사용
5. **독립성**: 각 문서는 단독으로 읽어도 맥락을 이해할 수 있어야 함

### 새 기능 문서 생성 절차

1. `docs/templates/`에서 필요한 템플릿을 `docs/features/{feature-name}/`으로 복사
2. 플레이스홀더(`{{...}}`)를 실제 값으로 교체
3. 라이프사이클 순서대로 작성: PRD → (구현) → TESTING → (검증) → TESTING-REPORT → PLAN

## 1. PRD (Product Requirements Document)

### 목적

기능의 요구사항을 LLM이 구현할 수 있는 수준으로 정의한다.

### 필수 섹션

| # | 섹션 | 내용 | 비고 |
|---|---|---|---|
| 1 | **개요** | 제품/기능 요약, 핵심 가치, 대상 사용자 | 3개 하위 섹션 필수 |
| 2 | **기능 요구사항** | 구현할 기능 목록과 상세 스펙 | 기능별 하위 섹션으로 분리 |
| 3 | **비기능 요구사항** | 브라우저 호환성, 반응형, 성능, 보안 등 | 해당 항목만 포함 |
| 4 | **데이터 스키마** | 저장/전송되는 데이터의 JSON 구조 | 코드블록으로 스키마 예시 |
| 5 | **기술 스택** | 사용 기술과 선택 이유 | 테이블 형식 |
| 6 | **제한사항** | 알려진 제약, 미지원 항목 | 불릿 리스트 |

### 작성 규칙

- **기능 요구사항**은 테이블로 구조화 (`| 항목 | 요구사항 |` 형식)
- **구현 범위**를 명확히: In Scope / Out of Scope를 구분하거나, 제한사항에서 명시적으로 제외 항목 나열
- **수치 기준 명시**: "빠르게 응답" → "10초 이내 응답", "충분히 큰" → "최소 44x44px"
- **데이터 스키마**는 실제 JSON 예시 포함 (주석으로 필드 설명)
- 번역/다국어 관련 기능은 지원 언어를 locale 코드로 명시 (예: `en-US`, `ja-JP`)

## 2. TESTING (Testing Guide)

### 목적

구현 결과를 수동 또는 LLM 기반으로 검증하기 위한 테스트 계획을 정의한다.

### 필수 섹션

| # | 섹션 | 내용 | 비고 |
|---|---|---|---|
| 1 | **사전 준비** | 테스트에 필요한 환경, 도구, 설정 | 불릿 리스트 |
| 2 | **테스트 시나리오** | 카테고리별 테스트 절차와 확인사항 | 번호 매긴 절차 + 체크리스트 |
| 3 | **에러 핸들링 검증** | 에러 시나리오별 예상 동작 | 테이블 형식 |

### 작성 규칙

- **PRD 대응**: PRD의 모든 기능 요구사항에 대응하는 테스트 시나리오가 존재해야 함
- **시나리오 구조**: 각 시나리오는 아래 형식을 따름:
  ```markdown
  ### N.M 시나리오 제목
  1. [동작 절차 1]
  2. [동작 절차 2]
  3. 확인사항:
     - [기대 결과 1]
     - [기대 결과 2]
  ```
- **재현 가능성**: 구체적인 입력값과 기대값 명시 (예: "Hello, how are you?" 입력 → `[EN]` 태그 표시)
- **카테고리 분류**: 관련 시나리오끼리 그룹화 (기본 동작, 제공자별, UI/UX 등)
- **에러 시나리오**는 `| 시나리오 | 예상 동작 |` 테이블로 정리

## 3. TESTING-REPORT (Quality Assessment)

### 목적

테스트 결과를 정량적으로 평가하고, 발견된 이슈를 분류하여 개선 우선순위를 제시한다.

### 필수 섹션

| # | 섹션 | 내용 | 비고 |
|---|---|---|---|
| 1 | **메타데이터** | 날짜, 프로젝트명, 기반 문서, 평가 범위 | 문서 상단 |
| 2 | **Overall Score** | 전체 점수 + 카테고리별 점수 테이블 | 10점 만점 |
| 3 | **카테고리별 평가** | 각 카테고리: 점수, Works well, Issues, Improvements | Severity 테이블 포함 |
| 4 | **Cross-cutting Concerns** | 보안, 접근성, 데이터 내구성, 코드 품질 등 횡단 이슈 | 해당 항목만 포함 |
| 5 | **Priority Roadmap** | P0/P1/P2 우선순위별 개선 항목 요약 | 번호 매긴 리스트 |

### 작성 규칙

- **스코어링 기준** (10점 만점):

  | 점수 | 등급 | 기준 |
  |---|---|---|
  | 9-10 | Excellent | 이슈 없음 또는 Low만 존재 |
  | 7-8 | Good | Medium 이하 이슈만 존재 |
  | 5-6 | Adequate/Needs Work | High 이슈 존재 |
  | 3-4 | Critical | Critical 이슈 존재 |
  | 1-2 | Broken | 핵심 기능 동작 불가 |

- **Severity 분류**:

  | Severity | 기준 | 대응 |
  |---|---|---|
  | **Critical** | 보안 취약점, 데이터 손실, 핵심 기능 장애 | 즉시 수정 (P0) |
  | **High** | 주요 기능 품질 저하, 사용성 심각 저해 | 우선 수정 (P0-P1) |
  | **Medium** | 사용성 불편, 비효율, 사소한 기능 누락 | 개선 권장 (P1-P2) |
  | **Low** | 코스메틱, 개선 제안, 미래 대비 | 선택적 (P2) |

- **이슈 테이블 형식**:
  ```markdown
  | Severity | Issue | Location |
  |----------|-------|----------|
  | Critical | [이슈 설명] | Line XXX |
  ```

- **코드 참조**: 이슈마다 코드 위치(Line 번호)를 명시하고, Critical/High 이슈는 문제 코드와 수정 코드를 함께 제시
- **카테고리 평가 구조**: 각 카테고리는 아래 형식을 따름:
  ```markdown
  ## N. 카테고리명 — X.X / 10

  **Works well:**
  - [잘 동작하는 항목]

  **Issues:**
  | Severity | Issue | Location |
  |----------|-------|----------|
  | ... | ... | ... |

  **Improvements:**
  - [개선 제안]
  ```

## 4. PLAN (Implementation Plan)

### 목적

TESTING-REPORT에서 발견된 이슈를 해결하기 위한 단계별 구현 계획을 수립한다.

### 필수 섹션

| # | 섹션 | 내용 | 비고 |
|---|---|---|---|
| 1 | **기반 문서 참조** | 기반이 되는 TESTING-REPORT 링크 + 현재 점수 | 문서 상단 |
| 2 | **Phase 구분** | P0 / P1 / P2 단계별 작업 항목 | Severity 기반 분류 |
| 3 | **각 작업 항목** | 문제 설명, 코드 위치, 해결 방법, 코드 예시 | Phase 내 번호 매기기 |
| 4 | **Implementation Order** | 전체 구현 순서 다이어그램 또는 리스트 | 의존성 고려 |
| 5 | **Out of Scope** | 이번 계획에서 제외하는 항목과 이유 | 불릿 리스트 |

### 작성 규칙

- **Phase 분류 기준**:

  | Phase | Severity 기반 | 설명 |
  |---|---|---|
  | Phase 1 (P0) | Critical + 일부 High | 보안, 데이터 무결성 등 즉시 수정 필요 |
  | Phase 2 (P1) | High + Medium | 신뢰성, 사용성 개선 |
  | Phase 3 (P2) | Medium + Low | 접근성, 폴리시, 부가 기능 |

- **작업 항목 구조**: 각 항목은 아래 형식을 따름:
  ```markdown
  ### N.M 항목 제목
  - **문제**: [TESTING-REPORT에서 식별된 이슈 요약]
  - **위치**: [코드 파일 + Line 번호]
  - **해결**:
    - [해결 방법 설명]
    - [코드 변경 예시 (코드블록)]
  ```

- **코드 예시**: 변경 전/후를 모두 보여주거나, 추가할 코드를 명확히 제시
- **의존성 명시**: 특정 작업이 다른 작업 완료 후에만 가능한 경우 명시
- **목표 점수**: 개선 후 예상 점수를 Phase별로 명시 (선택)
- **Out of Scope**: 제외 항목에는 반드시 이유를 함께 기술

## 체크리스트

새 기능 문서 작성 시 아래 체크리스트로 완성도를 확인한다:

### PRD 체크리스트
- [ ] 개요 3개 하위 섹션(제품 요약, 핵심 가치, 대상 사용자) 포함
- [ ] 모든 기능 요구사항이 테이블로 구조화
- [ ] 모호한 표현 없이 구체적 수치/조건 명시
- [ ] 데이터 스키마에 JSON 예시 포함
- [ ] 기술 스택 테이블 포함
- [ ] 제한사항/Out of Scope 명시

### TESTING 체크리스트
- [ ] PRD의 모든 기능 요구사항에 대응하는 테스트 존재
- [ ] 각 시나리오에 번호 매긴 절차 + 확인사항 포함
- [ ] 구체적인 입력값/기대값 명시
- [ ] 에러 시나리오 테이블 포함
- [ ] 사전 준비 섹션 포함

### TESTING-REPORT 체크리스트
- [ ] 메타데이터(날짜, 기반 문서 등) 포함
- [ ] Overall Score + 카테고리별 점수 테이블
- [ ] 모든 이슈에 Severity 레벨 + 코드 위치
- [ ] Critical/High 이슈에 코드 예시(문제/수정)
- [ ] Priority Roadmap 포함

### PLAN 체크리스트
- [ ] TESTING-REPORT 참조 링크 + 현재 점수
- [ ] Phase 3단계(P0/P1/P2) 구분
- [ ] 각 항목에 문제-위치-해결 구조
- [ ] 코드 변경 예시 포함
- [ ] Implementation Order 포함
- [ ] Out of Scope + 이유 포함
