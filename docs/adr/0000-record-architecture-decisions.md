# ADR 0000: ADR(아키텍처 결정 기록) 제도 도입

- **상태**: Accepted
- **날짜**: 2026-04-22
- **번호 체계**: 4자리 숫자 (`0000` ~ `9999`)

## 배경

프로젝트는 빠르게 변하고 있으며, "왜 이렇게 결정했는가"가 코드만으로는 드러나지 않는 경우가 늘고 있다. 코드 변경 이력(`git log`)은 **무엇이 변했는가**는 기록하지만 **왜 그 선택을 했는가**는 남기지 않는다. 논의가 충분히 이루어진 결정이 시간이 지나면 재논의·재번복되는 문제가 반복되었다.

## 결정

아키텍처/기술 선택에 영향을 주는 결정은 **ADR(Architecture Decision Record)** 문서로 남긴다.

### 위치 및 네이밍

- **디렉토리**: `docs/adr/`
- **파일명**: `NNNN-slug.md` (예: `0001-allow-js-css-separation.md`)
- **번호**: 4자리 숫자. 한 번 부여된 번호는 재사용하지 않는다
- **0000번**: 본 문서 — ADR 제도 자체에 대한 메타 결정

### 필수 섹션

```markdown
# ADR NNNN: 결정 제목

- **상태**: Proposed | Accepted | Deprecated | Superseded by ADR-XXXX
- **날짜**: YYYY-MM-DD

## 배경
왜 결정이 필요한가. 문제/제약.

## 결정
무엇을 결정했는가. 간결하고 단정적으로.

## 근거
왜 이 선택인가. 고려한 대안과 포기한 이유.

## 결과
결정으로 인해 생기는 변화/영향(긍정·부정 모두).
```

### 상태 전이

- `Proposed` → 논의 중
- `Accepted` → 적용
- `Deprecated` → 더 이상 권장하지 않음 (대체 ADR 없음)
- `Superseded by ADR-XXXX` → 다른 ADR로 대체됨 (기존 파일은 수정하지 않고 상태만 변경)

기존 ADR의 본문은 **수정하지 않는다**. 결정이 바뀌면 새 ADR을 만들고 이전 ADR의 상태를 `Superseded by ADR-XXXX`로 바꾼다.

## 근거

- **마이크로한 부담**: 프로젝트 규모상 본격적인 RFC 프로세스는 과하다. 파일 하나로 결정의 "배경/결정/근거"만 남겨도 충분하다
- **번호 4자리**: 3자리(최대 1000개)는 장기적으로 부족할 수 있고, 5자리는 과하다
- **0000을 메타 문서로 사용**: 후속 ADR이 "본 제도 자체를 어떻게 쓰는가"를 참고할 수 있도록 함

## 결과

- `docs/adr/` 디렉토리 생성
- 새로운 결정은 이 디렉토리에 ADR로 기록
- AGENTS.md 및 주요 규칙 문서에서는 ADR을 링크하고, 중복 설명하지 않는다
