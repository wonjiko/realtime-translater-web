# 서브에이전트 기반 검증 루프 — 설계 스펙

> 작성일: 2026-04-08
> 상태: Draft

---

## 1. 목적

피처 구현 완료 후, 독립된 컨텍스트의 서브에이전트들이 **점수 기반 검증 루프**를 수행하여 코드 품질을 보장한다. Pass/Fail이 아닌 1~10점 채점으로 "통과"뿐 아니라 "품질 향상"까지 유도한다.

## 2. 기존 워크플로우와의 관계

- **위치**: superpowers의 `brainstorming → writing-plans → executing-plans` 이후에 오는 **프로젝트 로컬 품질 게이트**
- **독립성**: superpowers와 별도. `docs/workflows/verification-loop.md`에 정의되고 `CLAUDE.md`에서 참조

## 3. 아키텍처

### 3-1. 서브에이전트 역할 (5개)

| 역할 | 투입 시점 | 입력 | 출력 |
|---|---|---|---|
| **플랜 에이전트** | 구현 시작 전 | 피처 요구사항 + 기존 코드 구조 | `{feature}-verification-plan.md` (초안) |
| **구현 에이전트** | 플랜 생성 후 | 플랜 + 스펙 | 코드 변경 |
| **플랜 보강 에이전트** | 구현 완료 후 | 초안 플랜 + git diff | `{feature}-verification-plan.md` (보강본) |
| **검증 에이전트** | 보강 완료 후 | 보강된 플랜 + 현재 코드 | `{feature}-verification-result.md` (점수) |
| **수정 에이전트** | 점수 미달 시 | 검증 결과 수정 가이드 + 현재 코드 | 코드 수정 |

### 3-2. 핵심 원칙

- 각 에이전트는 **독립된 컨텍스트**에서 실행 (서로의 대화 이력 없음)
- 에이전트 간 소통은 **오직 `docs/features/` 파일**을 통해서만
- 메인 세션은 **오케스트레이터** 역할만 수행 (파일 읽기 → 다음 에이전트 프롬프트 구성 → 디스패치)

### 3-3. 오케스트레이션 흐름

```
피처 요구사항 확정
  │
  ▼
① 플랜 에이전트 → verification-plan.md (초안)
  │
  ▼
② 구현 에이전트 → 코드 변경
  │
  ▼
③ 플랜 보강 에이전트 → verification-plan.md (보강본)
  │
  ▼
④ 검증 에이전트 → verification-result.md (Round N)
  │
  ▼
판정 확인 ─── 평균≥8 AND 최저≥5 ──→ ✅ 통과
  │
  │ FAIL
  ▼
정체 감지 ─── 평균 변화 ≤ 0.5 ──→ ⚠️ 사용자 개입 요청
  │
  │ 평균 변화 > 0.5
  ▼
⑤ 수정 에이전트 → 코드 수정
  │
  └──→ ④ 검증 에이전트 (다시)
```

## 4. 검증 플랜 구조

### 파일: `docs/features/{feature}-verification-plan.md`

```markdown
# {Feature Name} — 검증 플랜

## 메타
- 피처: {feature name}
- 생성: {date} (플랜 에이전트)
- 보강: {date} (플랜 보강 에이전트)

## 검증 카테고리

### 1. 기능 정합성 (Functional Correctness)
| # | 항목 | 검증 방법 | 10점 기준 | 5점 미만 기준 |
|---|------|----------|----------|-------------|
| 1-1 | ... | ... | ... | ... |

### 2. 에러 핸들링 (Error Handling)
(동일 테이블 구조)

### 3. 코드 품질 (Code Quality)
(동일 테이블 구조)

### 4. UX / 접근성 (UX & Accessibility)
(동일 테이블 구조)

### 5. 엣지 케이스 (Edge Cases)
(동일 테이블 구조)
```

- **카테고리 5개 고정**, 항목 수는 피처 규모에 따라 유동적
- **10점 기준 / 5점 미만 기준** 명시로 검증 에이전트의 채점 일관성 확보
- **검증 방법**: 보강 단계에서 실제 코드 위치(파일:라인)로 구체화됨

## 5. 검증 결과 구조

### 파일: `docs/features/{feature}-verification-result.md`

하나의 파일에 라운드별 섹션 누적.

```markdown
# {Feature Name} — 검증 결과

## 메타
- 피처: {feature name}
- 검증 플랜: {feature}-verification-plan.md
- 통과 기준: 평균 ≥ 8.0 AND 최저 ≥ 5

---

## Round N
- 검증 일시: {date}
- 검증 대상: {commit hash 또는 코드 상태}

### 점수표
| # | 카테고리 | 항목 | 점수 | 근거 |
|---|---------|------|------|------|
| 1-1 | 기능 정합성 | {항목명} | 7 | {1~2문장} |

### 카테고리 요약
| 카테고리 | 평균 | 최저 |
|---------|------|------|
| 기능 정합성 | 8.0 | 7 |

### 종합
- **전체 평균**: {N.N}
- **전체 최저**: {N} (항목 {X-Y})
- **판정**: ✅ PASS / ❌ FAIL
- **이전 대비**: 평균 {+/-N.N}, 최저 {+/-N} (Round 2부터)

### 수정 가이드 (FAIL인 경우)
1. **[필수] 항목 {X-Y} ({N}점)**: {개선 방법}
2. **[권장] 항목 {X-Y} ({N}점)**: {개선 방법}
```

- **근거** 컬럼: 수정 에이전트가 맥락 없이 이해할 수 있도록 필수
- **수정 가이드**: 5점 미만 `[필수]`, 8점 미만 `[권장]`
- **이전 대비** 델타: 오케스트레이터의 정체 감지에 사용

## 6. 통과 기준 & 정체 감지

### 통과 기준
- **전체 평균 ≥ 8.0** AND **모든 항목 ≥ 5**
- 두 조건 모두 충족 시 PASS

### 정체 감지
- **조건**: 현재 라운드 평균 - 이전 라운드 평균 ≤ 0.5
- **Round 1**: 정체 감지 건너뜀 (비교 대상 없음)
- **정체 시 사용자 선택지**:
  - 저점수 항목 직접 수정 후 검증 재실행
  - 해당 항목 통과 기준 완화
  - 현재 상태로 수용

## 7. 서브에이전트 프롬프트 템플릿

### ① 플랜 에이전트

```
역할: 피처 검증 플랜 설계자
목표: 주어진 피처 요구사항을 분석하여 검증 플랜을 생성한다.

## 입력
- 피처 요구사항: {요구사항 전문}
- 기존 코드 구조: {관련 파일 목록 및 주요 함수}

## 지시
1. 요구사항을 5개 카테고리(기능 정합성, 에러 핸들링, 코드 품질, UX/접근성, 엣지 케이스)로 분해
2. 카테고리별 검증 항목을 도출 (항목당 10점 기준 / 5점 미만 기준 명시)
3. 이 단계에서는 검증 방법을 "예상 구현 기준"으로 작성 (보강 단계에서 구체화됨)

## 출력
- docs/features/{feature}-verification-plan.md 파일 생성
- 포맷은 검증 플랜 구조 템플릿을 따를 것
```

### ② 구현 에이전트

```
역할: 피처 구현자
목표: 검증 플랜을 인지한 상태에서 피처를 구현한다.

## 입력
- 피처 요구사항: {요구사항 전문}
- 검증 플랜: {verification-plan.md 내용}

## 지시
1. 검증 플랜의 각 항목을 인지하고, 높은 점수를 받을 수 있도록 구현
2. 구현 완료 후 자체 점검: 플랜의 5점 미만 기준에 해당하는 부분이 없는지 확인
3. 코드 변경만 수행. 검증 플랜 파일은 수정하지 않을 것

## 출력
- 코드 변경 완료
- 변경 요약 (어떤 함수/파일을 추가/수정했는지)
```

### ③ 플랜 보강 에이전트

```
역할: 검증 플랜 보강자
목표: 실제 구현된 코드를 반영하여 검증 플랜의 검증 방법을 구체화한다.

## 입력
- 현재 검증 플랜: {verification-plan.md 내용}
- 코드 변경 diff: {git diff 결과}

## 지시
1. 각 검증 항목의 "검증 방법"을 실제 코드 위치(파일:라인)로 구체화
2. 구현 과정에서 생긴 새로운 검증 포인트가 있으면 항목 추가
3. 구현되지 않은 항목이 있으면 삭제하지 말고 그대로 유지 (검증 에이전트가 저점수 매김)
4. 메타 섹션의 "보강" 날짜 업데이트

## 출력
- docs/features/{feature}-verification-plan.md 업데이트
```

### ④ 검증 에이전트

```
역할: 독립 검증자
목표: 검증 플랜에 따라 현재 코드를 채점한다. 편향 없이 엄격하게 평가한다.

## 입력
- 검증 플랜: {verification-plan.md 내용}
- 검증 대상 코드: {관련 파일들의 현재 내용}
- 이전 라운드 결과 (있는 경우): {이전 라운드 종합 점수}

## 지시
1. 플랜의 모든 항목을 순서대로 검증
2. 각 항목에 1~10점 채점 + 근거(1~2문장) 기술
3. 10점 기준 / 5점 미만 기준을 참고하되, 자체 판단으로 중간 점수 산정
4. 카테고리 요약, 종합 판정 계산
5. FAIL인 경우 수정 가이드 작성: 5점 미만 [필수], 8점 미만 [권장]
6. 관대하게 채점하지 말 것. 의심스러우면 낮은 점수를 줄 것.

## 출력
- docs/features/{feature}-verification-result.md에 Round N 섹션 추가
```

### ⑤ 수정 에이전트

```
역할: 품질 개선자
목표: 검증 결과의 저점수 항목을 개선한다.

## 입력
- 검증 결과의 수정 가이드: {해당 라운드 수정 가이드 섹션}
- 검증 플랜: {verification-plan.md 내용}
- 현재 코드: {관련 파일들의 현재 내용}

## 지시
1. [필수] 항목을 먼저 처리 (5점 미만 → 5점 이상으로)
2. [권장] 항목을 이어서 처리 (8점 미만 → 8점 이상 목표)
3. 수정 시 다른 항목의 점수를 떨어뜨리지 않도록 주의
4. 검증 플랜, 검증 결과 파일은 수정하지 않을 것. 코드만 수정.

## 출력
- 코드 수정 완료
- 수정 내역 요약 (항목별로 무엇을 변경했는지)
```

## 8. 파일 구조

```
docs/
├── workflows/
│   └── verification-loop.md              ← 워크플로우 정의 (이 스펙 기반으로 작성)
├── features/
│   ├── {feature}-verification-plan.md    ← 플랜/보강 에이전트가 생성·수정
│   └── {feature}-verification-result.md  ← 검증 에이전트가 생성·누적
```

## 9. CLAUDE.md 연동

`CLAUDE.md`에 추가할 내용:

```markdown
## 검증 루프 워크플로우

피처 구현 시 `docs/workflows/verification-loop.md`의 프로세스를 따른다.

- 피처 구현이 완료되면 반드시 검증 루프를 실행할 것
- 검증 루프의 각 단계는 독립된 서브에이전트로 실행할 것
- 검증 통과 기준: 전체 평균 ≥ 8.0 AND 모든 항목 ≥ 5
- 결과 파일은 docs/features/ 에 저장
```
